## Фаза 1 — Безопасность (срочно)

### 1.1 `.dockerignore`
**Проблема:** `Dockerfile` делает `COPY . .` без `.dockerignore` — `.env`, `users.db` (пароли!), `.git`, `debug_dumps/`, шрифты, `timetable.json` запекаются в слои образа.
**Что сделать:** создать `.dockerignore` с `.env`, `users.db`, `users.db.bak*`, `.git/`, `debug_dumps/`, `__pycache__/`, `*.pyc`, `timetable.json`, `plans/`.
**Проверка:** пересобрать образ, убедиться что бот стартует (`.env`/`users.db` приходят через volume-mount — он уже настроен).

### 1.2 Шифрование паролей в `users.db`
**Проблема:** пароли от ЛК хранятся открытым текстом. Доступ к файлу = доступ ко всем учёткам студентов. Бот уже прощупывали SQL-инъекциями.
**Что сделать:**
- Добавить `cryptography` в `requirements.txt`, использовать `Fernet`.
- Ключ `ENCRYPTION_KEY` в `.env` (сгенерировать `Fernet.generate_key()`).
- Хелперы `encrypt_password()` / `decrypt_password()`.
- `perform_login` и авто-логин: шифровать при записи, расшифровывать при чтении.
- **Миграция:** разовый скрипт — сделать бэкап, прочитать существующие plaintext-пароли, перезаписать зашифрованными.
**Риск:** потеря `ENCRYPTION_KEY` = пароли не восстановить (юзерам придётся перелогиниться). Ключ хранить отдельно и надёжно.
**Примечание:** email не секрет — можно не шифровать.

### 1.3 Rate-limit на вход
**Проблема:** нет ограничения попыток `/login` — можно брутить пароли.
**Что сделать:** простой in-memory троттл — не более N попыток входа на `user_id` за окно времени; при превышении — «попробуй позже». Опционально — общий throttling-middleware aiogram на все сообщения.

## Фаза 2 — Надёжность

### 2.1 Ротация логов + лимит `debug_dumps`
**Проблема:** в `docker-compose.yml` блок `logging` закомментирован — логи растут безгранично. `debug_dumps/` пополняется HTML-дампами на каждый «нет кандидатов».
**Что сделать:**
- Включить в `docker-compose.yml` `logging: json-file, max-size 10m, max-file 3`.
- `debug_dumps` — хранить последние N файлов (чистить старые) либо отключать дампы флагом из `.env`.

### 2.2 Тесты на парсеры
**Проблема:** тестов нет вообще. Парсинг `lk.sut.ru`/`cabinet.sut.ru` хрупкий — сайт меняется, парсер ломается молча (как было с `get_schet`).
**Что сделать:**
- `pytest` + сохранённые HTML-фикстуры (raspisanie.php, raspisanie_all_new, message.php, subconto/search.php).
- Покрыть: `_parse_id_name_pairs`, разбор таблицы в `get_timetable`, `_get_week_safe` / `_get_week_param_safe`, regex в `lk_search_recipients`, парсинг `get_messages`.
- Небольшой рефактор: вынести чистый парсинг из сетевых функций (парсер принимает строку HTML).
**Польза:** при изменении сайта тест падает явно, а не «0 групп без ошибки».
