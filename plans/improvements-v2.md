## Фаза 1 — Эксплуатация и шум (срочно, 1–2 часа)

### 1.1 Debug-дампы: выключить в проде, почистить хвост

**Проблема:** `DEBUG_DUMPS` по умолчанию `"1"`. При каждом «нет кандидатов» для автоотметки пишется HTML. На диске уже **~950 файлов / 15 МБ** — вероятно накопление до внедрения `_prune_debug_dumps` или частые срабатывания при работающей автокликалке.

**Что сделать:**
- В `.env` на сервере: `DEBUG_DUMPS=0`, `DEBUG_DUMPS_KEEP=30` (на случай временной отладки).
- Изменить дефолт в коде: `os.getenv("DEBUG_DUMPS", "0")` — opt-in, не opt-out.
- Разово: `rm -rf debug_dumps/*.html` (или оставить 5–10 свежих для анализа).
- В README: когда включать дампы и как смотреть.

**Проверка:** после суток работы бота в `debug_dumps/` не появляются новые файлы при `DEBUG_DUMPS=0`.

### 1.2 Уровень логов из окружения

**Проблема:** в `main.py` для root и `aiogram` задан `logging.DEBUG` — в проде логи раздуваются, возможны лишние данные в логах.

**Что сделать:**
- `LOG_LEVEL=INFO` в `.env` (дефолт INFO).
- `logging.basicConfig(level=getattr(logging, os.getenv("LOG_LEVEL", "INFO")))`.
- `aiogram` — не ниже WARNING в проде, если не отладка.

**Проверка:** `docker compose logs` без DEBUG-спама при обычной работе.

### 1.3 Healthcheck и корректная остановка

**Проблема:** в Docker нет проверки «бот жив»; при `restart` фоновые задачи (`auto_login_all_users`, `LessonController`) могут обрываться грубо.

**Что сделать:**
- Файл-метка «polling запущен» или простой HTTP health на localhost (опционально).
- В `docker-compose.yml`: `healthcheck` (например, проверка процесса / файла heartbeat).
- Обработчик shutdown: остановить контроллеры, дождаться текущих LK-запросов (timeout 10–15 с).

**Проверка:** `docker compose restart` — бот поднимается, пользователи не получают дубликатов «автоотметка запущена».

## Фаза 2 — Надёжность и регрессии

### 2.1 CI: автоматический pytest

**Проблема:** 122 теста есть, но без CI легко сломать парсер при правке `main.py` и не заметить.

**Что сделать:**
- GitHub Actions / GitLab CI на **Python 3.12** (как в `Dockerfile`, а не 3.11).
- Установка зависимостей: `pip install -r requirements-dev.txt` — **не** `requirements.txt`:
  `pytest` / `pytest-cov` лежат только в dev-файле (он уже включает `-r requirements.txt`).
- ⚠️ **Тесты импортируют `main.py`, а он при импорте создаёт `Bot(token=...)` и читает
  `ENCRYPTION_KEY`.** Без переменных окружения сбор тестов падает ещё до запуска —
  `TokenValidationError` (проверено). В workflow задать env:
  - `BOT_TOKEN` — фиктивный, но валидного формата (например `123456:AAaaBBbbCCccDDdd...`);
    сетевых запросов при импорте нет, токен только проходит формат-валидацию aiogram;
  - `ENCRYPTION_KEY` — сгенерировать прямо в шаге workflow:
    `python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"`
    (иначе падает тест `test_real_encryption_key_is_valid`).
- Запуск: просто `pytest` — coverage уже прописан в `pytest.ini`.
- Кэш pip; артефакт coverage (опционально); badge в README.
- Примечание: потребность в env-переменных уйдёт после **4.1** (вынос `config.py` /
  `on_startup` — импорт `main` перестанет иметь side-effects). До тех пор — через env CI.

**Проверка:** PR с намеренно сломанным `parse_week_number` — пайплайн красный; чистый PR — зелёный.

### 2.2 Зафиксировать версии зависимостей

**Проблема:** в `requirements.txt` пакеты без пинов (кроме `pillow>=10`). Обновление `bonchapi` / `aiogram` может сломать бота без изменений в репо.

**Что сделать:**
- `pip freeze` в `requirements.lock` или верхние границы: `aiogram==3.x.y`, `bonchapi==…`.
- Запинить и `requirements-dev.txt` (`pytest`, `pytest-cov`) — чтобы CI был воспроизводим.
- Раз в месяц — осознанное обновление с прогоном тестов.

### 2.3 Мониторинг «сломался парсер»

**Проблема:** при смене вёрстки lk.sut.ru бот может молча не находить кнопки / недели — пользователь видит только «клик не выполнен».

**Что сделать:**
- Счётчик ошибок парсинга за час (in-memory или SQLite).
- При пороге (например, >10 `no_candidates` за 30 мин) — одно сообщение админу (`ADMIN_IDS` в `.env`).
- Не слать админу полный HTML — только счётчик и `user_id` (опционально).

**Зависимости:** проще после 2.1 и 1.1 (меньше шума от дампов).

## Фаза 3 — UX (видимая польза для студентов)

### 3.1 Кэш `timetable.json` с TTL

**Проблема:** при старте и по `/reload_timetable` тянется расписание всех групп — долго и нагружает cabinet.sut.ru. Файл есть, но политика обновления неочевидна.

**Что сделать:**
- Метаданные в JSON: `fetched_at`, `ttl_hours` (например 6).
- При запросе группы: если кэш свежий — отдавать сразу; фоном — optional refresh.
- Кнопка «🔄 Обновить расписание групп» в меню расписания (уже есть reload — привязать к TTL в тексте: «обновлено N мин назад»).

### 3.2 Быстрые пресеты расписания

**Что сделать:**
- Inline-кнопки: «Сегодня», «Завтра», «Текущая неделя» для «Моё расписание» и группы.
- Использовать уже существующие `format_timetable_dict` / навигацию по неделям — без нового парсера.

### 3.3 Кэш списка сообщений

**Проблема:** повторный заход в «Сообщения» снова тянет все страницы ЛК.

**Что сделать:**
- `message_states` + timestamp; при повторном входе < 3–5 мин — показать кэш с кнопкой «Обновить».
- Инвалидация после отправки сообщения.

## Фаза 4 — Архитектура кода (без спешки)

### 4.1 Декомпозиция `main.py` (продолжение 4.2 из v1)

**Проблема:** ~4600 строк, сложно ревьюить и тестировать интеграционно.

> `parsers.py` уже вынесен в v1 — чистые парсеры HTML отделены от сети и покрыты тестами.
> Эта задача продолжает разбиение остального кода.

**Безопасная последовательность** (каждый шаг — отдельный коммит + pytest):

1. `config.py` — env, константы, `LESSON_INTERVALS`, логирование. Сюда же — отложенная
   инициализация `Bot`/БД, чтобы `import main` перестал иметь side-effects (см. 2.1).
2. `db.py` — SQLite, миграции колонок, `get/set_notify_*`.
3. `security.py` — Fernet, rate-limit.
4. `keyboards.py` + `states.py` — UI без логики.
5. `lk_client.py` — `DebuggableBonchAPI`, `lk_*`.
6. `handlers/` — по файлу: `schedule`, `messages`, `profile`, `autoclick`.
7. `main.py` — только `main()`, `on_startup`, регистрация роутеров.

**Не трогать в том же PR:** поведение автокликалки, формат сообщений пользователю.

### 4.2 Один клиент для ЛК

**Проблема:** дублирование — `bonchapi.BonchAPI`, `DebuggableBonchAPI`, `TImetabels.BonchAPI` (логин, cookies, заголовки).

**Что сделать:** общий базовый класс или адаптер с единым `BROWSER_HEADERS` и cookie jar; `TImetabels` оставить только публичное расписание без логина.

**Риск:** регрессия авторизации — только после 2.1 и расширения фикстур.

### 4.3 Тесты на критичный путь автоотметки

**Что покрыть фикстурами:**
- `click_start_lesson` при `raspisanie_with_lessons.html` → clicked > 0.
- `no_candidates` → 0, без исключения.
- `get_upcoming_start_lesson_details` для напоминания.

Сейчас логика пар частично в `test_main_lesson_logic.py` — добавить сценарии с реальными кусками HTML из `debug_dumps/` (обезличить).
