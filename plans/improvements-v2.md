## Фаза 1 — Эксплуатация и шум (срочно, 1–2 часа)

### 1.1 Debug-дампы: opt-in вместо opt-out

**Состояние:** `_prune_debug_dumps` (внедрён в v1) работает — папка держится в пределах
`DEBUG_DUMPS_KEEP=30` (сейчас ~30 файлов / 0.5 МБ, не растёт). Острой проблемы с диском
**больше нет** — прежние ~950 файлов почистились автоматически. Осталась мелочь: дампы
по умолчанию **включены** (`DEBUG_DUMPS` дефолт `"1"`), то есть в проде HTML-дампы
страниц ЛК пишутся без явного запроса.

**Что сделать:**
- Изменить дефолт в коде: `os.getenv("DEBUG_DUMPS", "0")` — opt-in, не opt-out.
- В `.env` на сервере явно `DEBUG_DUMPS=0` (или убрать переменную — дефолт станет 0).
- В README/`.env.example`: когда включать дампы (`DEBUG_DUMPS=1`) и где их смотреть.

**Проверка:** после суток работы при `DEBUG_DUMPS=0` новых файлов в `debug_dumps/` нет.

### 1.2 Уровень логов из окружения

**Проблема:** в `main.py` для root и `aiogram` задан `logging.DEBUG` — в проде логи раздуваются, возможны лишние данные в логах.

**Что сделать:**
- `LOG_LEVEL=INFO` в `.env` (дефолт INFO).
- `logging.basicConfig(level=getattr(logging, os.getenv("LOG_LEVEL", "INFO")))`.
- `aiogram` — не ниже WARNING в проде, если не отладка.

**Проверка:** `docker compose logs` без DEBUG-спама при обычной работе.

### 1.3 Healthcheck и корректная остановка

**Проблема:** в Docker нет проверки «бот жив»; при `restart` фоновые задачи (`auto_login_all_users`, `LessonController`) могут обрываться грубо.

**Что сделать:**
- Файл-метка «polling запущен» или простой HTTP health на localhost (опционально).
- В `docker-compose.yml`: `healthcheck` (например, проверка процесса / файла heartbeat).
- Обработчик shutdown: остановить контроллеры, дождаться текущих LK-запросов (timeout 10–15 с).

**Проверка:** `docker compose restart` — бот поднимается, пользователи не получают дубликатов «автоотметка запущена».

## Фаза 2 — Надёжность и регрессии

### 2.1 CI: автоматический pytest — ❌ ОТМЕНЁН

**Решение (2026-05-18):** не делаем. Деплой идёт через **Coolify** по push в `main`;
GitHub Actions запускается параллельно и деплой **не блокирует** — Coolify собирает
и катит образ независимо от результата CI. Пуш идёт прямо в `main` (без PR-flow),
поэтому CI был бы лишь уведомлением постфактум «выкатил сломанное», а не гейтом.

**Что вместо этого:** гонять `pytest` локально перед `git push`. Если в будущем
понадобится настоящий гейт под Coolify — добавить тест-шаг в `Dockerfile`
(`RUN pytest`): упавший тест → упавшая сборка → Coolify не деплоит, старый
контейнер остаётся жив. Тогда пригодятся заметки ниже про env.

> Полезные детали, если CI/гейт всё-таки понадобится:
> - Зависимости: `pip install -r requirements-dev.txt` (включает `-r requirements.txt`;
>   `pytest`/`pytest-cov` — только в dev-файле).
> - ⚠️ Тесты импортируют `main.py`, а он при импорте создаёт `Bot(token=...)` и читает
>   `ENCRYPTION_KEY`. Нужны env-переменные:
>   - `BOT_TOKEN` — фиктивный валидного формата (`123456:любая-непустая-строка`);
>     сетевых запросов при импорте нет, идёт только формат-валидация aiogram;
>   - `ENCRYPTION_KEY` — сгенерировать: `python -c "from cryptography.fernet import
>     Fernet; print(Fernet.generate_key().decode())"` (иначе падает
>     `test_real_encryption_key_is_valid`).
> - Потребность в env уйдёт после **4.1** (вынос `config.py` — импорт `main`
>   перестанет иметь side-effects).

### 2.2 Зафиксировать версии зависимостей — ✅ СДЕЛАНО

**Проблема:** в `requirements.txt` пакеты без пинов (кроме `pillow>=10`). Обновление `bonchapi` / `aiogram` может сломать бота без изменений в репо.

**Что сделано:**
- Все прямые зависимости запинены `==` (текущие рабочие версии).
- В `requirements.txt` добавлены `aiofiles` / `aiohttp` / `yarl` — `main.py` импортирует
  их напрямую, но раньше они тянулись только транзитивно через `aiogram`.
- `requirements-dev.txt` (`pytest`, `pytest-cov`) тоже запинены.
- Раз в месяц — осознанное обновление с прогоном тестов (`pytest`).

### 2.3 Мониторинг «сломался парсер» — ✅ СДЕЛАНО

**Проблема:** при смене вёрстки lk.sut.ru бот может молча не находить кнопки / недели — пользователь видит только «клик не выполнен».

**Что сделано:**
- Сигналом поломки выбран **`no_week_param`**, а не `no_candidates`: на валидной
  странице расписания `week_param` есть всегда, тогда как `no_candidates` —
  норма (пара уже отмечена → кнопки нет), и его счётчик давал бы ложные алерты.
- `ParserFailureMonitor` — скользящее окно, считает сбои по **разным**
  пользователям (один юзер, циклящий каждую минуту, алерт не вызовет).
- При пороге (≥3 разных юзеров за 30 мин) — одно сообщение всем `ADMIN_IDS`,
  дальше cooldown 60 мин. В тексте — только счётчик, без HTML.
- Env: `ADMIN_IDS`, `PARSER_ALERT_WINDOW_MIN/THRESHOLD/COOLDOWN_MIN`.
- Чистая логика (`ParserFailureMonitor`, `_parse_admin_ids`) покрыта тестами
  (`tests/test_parser_monitor.py`).

## Фаза 3 — UX (видимая польза для студентов)

### 3.1 Кэш `timetable.json` с TTL — ✅ СДЕЛАНО

**Проблема:** при старте и по `/reload_timetable` тянется расписание всех групп — долго и нагружает cabinet.sut.ru. Файл есть, но политика обновления неочевидна.

**Что сделано:**
- Метка времени загрузки — в sidecar-файле `timetable.meta.json` (`fetched_at`).
  Сам `timetable.json` (~58 МБ) и его формат не трогали — нулевой риск для
  всех потребителей кэша.
- `TIMETABLE_TTL_HOURS` (env, дефолт 6). При запросе группы: если кэш в памяти
  старше TTL и фоновая загрузка не идёт — `_refresh_timetable_quietly()`
  обновляет его в фоне, пользователю сразу отдаются текущие данные.
- Кнопка «🔄 Обновить расписание групп» в меню расписания; в тексте меню —
  «🗂 Расписание групп обновлено: N мин/ч назад».
- Чистые хелперы (`_timetable_age_seconds`, `_is_timetable_stale`,
  `_format_cache_age` и т.д.) покрыты тестами — `tests/test_timetable_cache.py`.

### 3.2 Быстрые пресеты расписания — ✅ СДЕЛАНО

**Что сделано:**
- Inline-кнопки «📍 Сегодня» / «Завтра 📍» добавлены в навигацию недель —
  и для «Моё расписание» (`get_week_navigation_buttons`), и для группы
  (`get_group_week_navigation_buttons`). «Текущая неделя» уже была.
- Обработчики `process_my_day` (offset 0/1) и `process_group_day`.
  Группа фильтруется по дате прямо из дикт-кэша; личное расписание — через
  `_week_offset_for_date` (корректно ловит «завтра», уходящее в след. неделю).
- Чистые хелперы `filter_group_lessons_by_date`,
  `filter_personal_lessons_by_date`, `_week_offset_for_date` покрыты тестами
  (`tests/test_timetable_presets.py`). Парсер не трогали — фильтр поверх
  существующего `format_timetable` / `format_timetable_dict`.

### 3.3 Кэш списка сообщений — ✅ СДЕЛАНО

**Состояние:** в v1 `/messages` переведён на **ленивую загрузку** — грузится только
1-я страница (~20 свежих сообщений, ~2–3 с), остальные подтягиваются при листании
«Вперёд». Прежняя боль («тянет все ~35 страниц ~10 с») закрыта.

**Что сделано:**
- Счётчик сообщений: вместо «20+» — оценка `≈N` (страниц × размер страницы);
  `format_message_count` хранит `per_page` в `message_states`.
- Тёплый кэш: повторный заход в `/messages` в пределах `MESSAGES_CACHE_TTL_SEC`
  (300 с) показывает уже загруженное без перезапроса первой страницы из ЛК.
  Кнопка «🔄 Обновить список» остаётся; кэш сбрасывается после отправки
  сообщения (`_invalidate_messages_cache`).
- Чистые хелперы `format_message_count`, `_messages_cache_fresh` покрыты
  тестами (`tests/test_messages_cache.py`).

## Фаза 4 — Архитектура кода (без спешки)

### 4.1 Декомпозиция `main.py` (продолжение 4.2 из v1)

**Проблема:** ~4600 строк, сложно ревьюить и тестировать интеграционно.

> `parsers.py` уже вынесен в v1 — чистые парсеры HTML отделены от сети и покрыты тестами.
> Эта задача продолжает разбиение остального кода.

**Безопасная последовательность** (каждый шаг — отдельный коммит + pytest):

1. `config.py` — env, константы, `LESSON_INTERVALS`, логирование. Сюда же — отложенная
   инициализация `Bot`/БД, чтобы `import main` перестал иметь side-effects (сейчас
   тесты вынуждены подсовывать фиктивные `BOT_TOKEN`/`ENCRYPTION_KEY`).
2. `db.py` — SQLite, миграции колонок, `get/set_notify_*`.
3. `security.py` — Fernet, rate-limit.
4. `keyboards.py` + `states.py` — UI без логики.
5. `lk_client.py` — `DebuggableBonchAPI`, `lk_*`.
6. `handlers/` — по файлу: `schedule`, `messages`, `profile`, `autoclick`.
7. `main.py` — только `main()`, `on_startup`, регистрация роутеров.

**Не трогать в том же PR:** поведение автокликалки, формат сообщений пользователю.

### 4.2 Один клиент для ЛК

**Проблема:** дублирование — `bonchapi.BonchAPI`, `DebuggableBonchAPI`, `TImetabels.BonchAPI` (логин, cookies, заголовки).
