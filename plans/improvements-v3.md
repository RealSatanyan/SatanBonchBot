## Часть A — Архитектура и «красивый репозиторий»

### A.1 Один клиент для ЛК (бывш. 4.2)

**Проблема:** дублируются три клиента к sut.ru — `bonchapi.BonchAPI` (внешняя
либа), `DebuggableBonchAPI` (наш, `lk_client.py`), `TImetabels.BonchAPI`
(публичное расписание, `TImetabels.py`). У каждого свои логин, cookie jar,
заголовки, ретраи.

**Что сделать:**
- Общий базовый слой: единый `BROWSER_HEADERS`, общий cookie jar, общий
  http-хелпер с ретраями (сейчас ретраи есть только в `TImetabels`).
- `TImetabels` оставить ТОЛЬКО публичное расписание без логина; авторизованную
  работу с ЛК свести к одному классу в `lk_client.py`.
- Свести разнобой User-Agent / заголовков к одному источнику.

**Риск:** регрессия авторизации и автоотметки. Делать только под защитой тестов
из `test_click_lesson.py` (4.3) — и сначала расширить их фикстуры (сценарии
логина, переавторизации). **Сначала A.4 в части LK-клиента, потом A.1.**

**Проверка:** автоотметка и все три раздела расписания работают; `pytest` зелёный.

### A.2 Убрать мёртвый код

**Что нашлось при декомпозиции 4.1:**
- `SendMsgAPI.py` — отдельный скрипт-прототип, нигде не импортируется, читает
  несуществующий `options.txt`. Его логика уже перенесена в `lk_client.py`
  (`lk_upload_file`, `lk_send_message`). **Удаление НЕ здесь** — это финальный
  шаг B.2: пока фича загрузки файлов не доделана, скрипт держим как референс.
- `TImetabels.py` — содержит CLI-интерфейс (`wait_key`, `cls`,
  `*_interface`, блок `if __name__ == '__main__'`), не нужный боту. Вырезать
  CLI, оставить парсинг/загрузку.
- `migrate_passwords.py` — одноразовый скрипт миграции (выполнен в v1).
  Удалить или переместить в `scripts/` с пометкой «уже выполнено».
- Неиспользуемые импорты в модулях после 4.1 — пройтись линтером
  (`ruff`/`pyflakes`), почистить.
- `plans/4.1-analysis/` — 6 промежуточных файлов анализа. Архивировать или
  удалить (итог — в `4.1-decomposition.md`).

**Проверка:** `pytest` зелёный; `import main` без ошибок; репозиторий без
файлов-сирот.

### A.3 Документация структуры

**Что сделать:**
- `ARCHITECTURE.md` — карта модулей и слоёв (config → db/security →
  lk_client/lesson_controller → сервисы → handlers → main), правила

**Что сделать:**

- `rendering.py` — хотя бы smoke-тест (картинка генерится, файл не пустой).
- Цель: сервисы и критичные хэндлеры — выше 60%.

## Часть B — Фичи

### B.1 Полные ФИО преподавателей + корпус

**Хочу:** показывать полное ФИО преподавателя (если доступно) и корпус уника,
а не только «Иванов И.И.» / «ауд. 401».

**⚠️ Шаг 0 — разведка (обязательно перед оценкой).** Сейчас парсер
(`parsers.parse_timetable_table`) берёт `span.teacher` и `span.aud` как есть.
Тестовая фикстура `group_timetable.html` обезличена («Иванов И.И.», «ауд. 401»)
— по ней НЕ видно, есть ли в реальном HTML `cabinet.sut.ru/raspisanie_all_new`:
- полное ФИО (в `title`-атрибуте, вложенном теге, или ссылке на препода);
- номер корпуса (в тексте `span.aud`, отдельном элементе, или выводится из
  диапазона номеров аудиторий).

**Что сделать:**
1. Снять реальный HTML расписания группы с `cabinet.sut.ru`, посмотреть глазами.
2. Если полное ФИО / корпус есть в разметке — расширить
   `parse_timetable_table`: добавить поля `'ФИО преподавателя (полное)'`,
   `'Корпус'`; обновить фикстуру (реальным обезличенным примером) и тесты
   `test_parsers.py`.
3. Если корпус не размечен, но выводится из номера аудитории — таблица
   соответствий «диапазон номеров → корпус».
4. Обновить отображение: `formatting.format_timetable_dict` и
   `rendering.generate_timetable_image_from_dict` — показывать полное ФИО
   (с фолбэком на краткое) и корпус.

**Риск:** зависит от разведки — если данных в HTML нет, фича сужается или
требует доп. источника (справочник преподавателей). Объём M (если данные есть)
или L (если нужен отдельный источник).

### B.2 Загрузка своих файлов в ЛК

**Хочу:** прикреплять свой файл к сообщению в ЛК.

**Состояние:** механизм загрузки уже есть — `lk_client.lk_upload_file`
(POST `message_create_stud.php`, FormData, возвращает `idinfo`; реализован по
образцу `SendMsgAPI.upload_file`). НО flow «✏️ Написать» вызывает
`lk_send_message` всегда с `idinfo=0` — прикрепить файл нельзя.

**Что сделать:**
- Новое FSM-состояние `write_file` (после `write_text`): «прикрепить файл или
  пропустить».
- Хэндлер на `types.Message` с `document`/`photo`: скачать файл из Telegram во
  временный файл, вызвать `lk_upload_file`, получить `idinfo`.
- Передать `idinfo` в `lk_send_message` вместо `0`.
- Ограничения: размер (лимит Telegram — 20 МБ на bot-download), чистка temp,
  понятная ошибка при сбое загрузки.
- **Финальный шаг:** фича готова и проверена → референс больше не нужен →
  удалить `SendMsgAPI.py`.

**Проверка:** отправка сообщения с вложением доходит в ЛК; без вложения —
работает как раньше.

### B.3 Объединять группы в расписании препода/аудитории

**Проблема:** `teacher_timetable` / `classroom_timetable` собирают занятия по
ВСЕМ группам. Лекция-поток для 5 групп = 5 почти одинаковых строк, разница
только в поле `Группа`. Длинно и неудобно.

**Что сделать:**
- Шаг дедупликации: занятия с одинаковыми (неделя, день, время, предмет, тип,
  преподаватель, аудитория) и разными группами — слить в одну запись с
  `'Группы': ['А', 'Б', 'В']`.
- `format_timetable_dict` (и рендер картинкой) — показывать «👥 Группы: А, Б, В»
  вместо одной группы, когда их несколько.
- Чистая логика (функция дедупа поверх списка занятий) — покрыть тестами.

**Проверка:** расписание препода-потоковика — одна строка на пару со списком
групп; расписание одной группы не меняется.

## Часть C — На будущее (уведомления)

### C.0 Предусловие — привязка пользователь → группа

Для C.1 боту нужно знать группу пользователя. Сейчас он её не хранит явно
(«Моё расписание» идёт через личный ЛК; группа в кэше — по имени, вводится
руками). Понадобится: определять/хранить группу пользователя (из ЛК при логине
или явной настройкой). Колонка `group_name` в `users` (миграция как в db.py).

### C.1 Уведомления об изменении расписания

**Идея:** бот и так периодически перезагружает `timetable.json` всех групп.
При обновлении — сравнить новый снимок группы со старым, найти диффы (добавлена/
убрана/перенесена пара), уведомить пользователей этой группы.

**Подводные камни:** что считать «изменением» (сервер может тасовать порядок);
не спамить; диффать аккуратно. Объём L–XL, риск высокий. Зависит от C.0.

### C.2 Уведомления о новых сообщениях в ЛК

**Идея:** периодически (фоновая задача, как `auto_login_all_users`) опрашивать
1-ю страницу входящих каждого авторизованного пользователя, сравнивать с
последним известным `message_id`, при новом — уведомлять.

**Подводные камни:** частота опроса vs нагрузка на ЛК и антибот; хранить
«последний виденный id» (колонка в `users`); не дублировать уведомления.
Объём L, риск средний.

## Рекомендованный порядок работ

```
Сначала «красивый репозиторий» (быстро, низкий риск):
  A.2 (мёртвый код, КРОМЕ SendMsgAPI.py) → A.3 (доки) → A.4 (тесты)

Архитектура (под защитой тестов A.4):
  A.1 (один клиент для ЛК)

Фичи (видимая польза):
  B.2 (файлы — почти готово; финал B.2 удаляет SendMsgAPI.py)
  → B.3 (объединение групп) → B.1 (ФИО+корпус, начать с разведки HTML)

Будущее, по мере желания:
  C.0 → C.1 ; C.2
```

**Логика порядка:** B.2 почти готов (механизм есть, нужен UI) — быстрый
выигрыш. `SendMsgAPI.py` удаляется именно в финале B.2 — это его референс,
до готовности фичи скрипт не трогаем. B.1 начинать с разведки: пока не снят
реальный HTML, объём неизвестен. A.1 — самый рискованный, делать после A.4
(расширенные тесты LK-клиента как страховка).
